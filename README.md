В проекте реализованы следующие промежуточные обработчики:

1. Логирование запросов  
   Выводит HTTP-метод, путь и статус ответа

   Добавляет заголовок:

```

```

3. Проверка ключа доступа  
   Проверяет параметр `?key=secret`, при отсутствии возвращает 401

## Ответы на вопросы

### 1. Minimal API и MVC

Minimal API — это упрощённый способ создания веб-API без использования контроллеров и сложной архитектуры. Вся логика обычно находится в Program.cs, а маршруты задаются напрямую через app.MapGet, app.MapPost и другие методы.

Преимущества Minimal API:

- минимальное количество кода
- быстрый старт разработки
- подходит для микросервисов и небольших проектов

Недостатки:

- сложнее поддерживать при росте проекта
- отсутствует чёткая структура

MVC (Model-View-Controller) — это архитектурный паттерн, который разделяет приложение на три части: модель, представление и контроллер.

Преимущества MVC:

- хорошая масштабируемость
- чёткое разделение ответственности
- удобство поддержки больших проектов

Недостатки:

- больше кода и настроек
- сложнее для начинающих

---

### 2. Порядок middleware

Если вызвать app.MapGet(...) до app.Use(...), то middleware не будет применяться к этим маршрутам.

Это происходит потому, что middleware обрабатывают запросы в том порядке, в котором они добавлены в конвейер. Если маршрут зарегистрирован раньше, он может завершить обработку запроса, и до middleware выполнение не дойдёт.

Правильный порядок:

- сначала app.Use(...)
- затем app.MapGet(...)

Это гарантирует, что все запросы пройдут через middleware.

---

### 3. Почему используется camelCase

ASP.NET Core по умолчанию преобразует имена свойств из PascalCase в camelCase при сериализации JSON.

Причины:

- в JavaScript принято использовать camelCase для имен переменных и свойств
- большинство frontend-фреймворков (React, Angular, Vue) ожидают именно такой формат
- это стало фактическим стандартом при работе с JSON API

Такое преобразование упрощает взаимодействие между backend (C#) и frontend (JavaScript).

---

### 4. HTTP-коды

HTTP-коды используются для обозначения результата обработки запроса сервером.

Основные коды:

- 200 OK
  Запрос успешно обработан, сервер вернул корректный ответ

- 401 Unauthorized
  Клиент не прошёл аутентификацию. Требуется предоставить учетные данные (например, токен или логин/пароль)

- 404 Not Found
  Запрашиваемый ресурс не найден. Сервер работает, но такого маршрута или объекта не существует

Дополнительно:

- 403 Forbidden — доступ запрещен, даже если пользователь авторизован
- 500 Internal Server Error — ошибка на стороне сервера
